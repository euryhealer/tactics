1. Vision del Motor

El proyecto puede evolucionar hacia un motor 2.5D isometrico basado en tiles, footprints y escenas conectadas. La idea central es separar el contenido del juego de los sistemas generales: terreno, camara, colocacion, inventario, economia, persistencia, escenas interiores y herramientas de editor.

La gran ventaja es que el mismo sistema de bloques puede servir para mundos exteriores, interiores, bases, ciudades, dungeons, granjas, tableros tacticos o mapas editables.

4. Contratos de Datos

Para convertir esto en motor, conviene formalizar los datos. Los siguientes contratos son la forma recomendada de pensar el save y la comunicacion futura con backend.

Tile

{
  "x": 0,
  "y": 0,
  "type": "grass",
  "height": 1,
  "crop": null
}

Building

{
  "id": "house",
  "x": 8,
  "y": 1,
  "width": 2,
  "height": 2,
  "interiorId": "house"
}

Interior

{
  "id": "house",
  "items": {
    "bed": { "x": 1, "y": 1 },
    "stove": { "x": 5, "y": 1 },
    "wardrobe": { "x": 0, "y": 1 }
  }
}

Item Stack

{
  "id": "carrot",
  "qty": 12
}

5. Ruta Hacia Backend

El paso correcto no es conectar backend directo desde cada escena. Conviene introducir una capa de servicios entre estado local y red.

Paso Objetivo Archivo/capa sugerida
1 Crear un adapter de save/load con la misma API para localStorage y backend. src/services/save-service.js
2 Centralizar serializacion/deserializacion del perfil. src/core/profile-save.js
3 Agregar estado de sesion, userId y token. src/services/session-service.js
4 Sincronizar cambios importantes por eventos: mover building, plantar, vender, editar interior. src/services/sync-service.js
5 Resolver conflictos entre cliente y servidor. Versiones de save, timestamps, revision number.

Para multiplayer o persistencia online, el servidor deberia validar acciones de economia, construccion y movimiento de objetos importantes. El cliente puede seguir renderizando y previsualizando, pero el backend deberia ser autoridad en saves compartidos.

6. Editores y Herramientas

El perfil GM ya funciona como modo editor. Eso puede crecer hacia herramientas internas:

  • Editor de terrain brushes: grass, soil, water, altura, erase.
  • Editor de buildings con catalogo, footprint y reglas de colocacion.
  • Editor de interiores con catalogo de muebles.
  • Editor de crops, timers, precios y drops.
  • Export/import de mapas como JSON.
  • Validadores de mapa: areas desconectadas, edificios fuera de bounds, objetos bloqueando salidas.

8. Fortalezas del Sistema de Bloques Isometricos

  • Permite mucha profundidad visual con sprites 2D livianos.
  • El grid da reglas claras para colocar, mover, validar y persistir objetos.
  • Las alturas agregan lectura tactica sin necesitar 3D real.
  • El mismo renderer sirve para exteriores e interiores.
  • Los footprints unifican edificios, muebles, maquinas y unidades grandes.
  • La rotacion de camara en 4 angulos mejora inspeccion de mapas densos.

9. Limites y Riesgos

Riesgo Impacto Solucion recomendada
Estado global compartido Puede crecer dificil de mantener. Mover a stores/servicios con API clara antes de backend.
Render inmediato con graphics Mapas grandes pueden bajar performance. Chunking, dirty rendering o tilemap/cache por capas.
Pathing basico No escala a NPCs complejos. A* con costos por terreno, bloqueo dinamico y reservas.
Persistencia local No sirve para cuentas reales o multi-dispositivo. Save service con backend y migraciones versionadas.
Catalogos hardcoded Agregar contenido requiere tocar codigo. Mover crops, items, buildings y furniture a JSON.

10. Roadmap Recomendado

  1. Extraer catalogos de contenido a JSON: items, crops, buildings, furniture, blocks.
  2. Crear save-service con backend/localStorage intercambiables.
  3. Crear un sistema generico de escenas interiores por interiorId.
  4. Unificar placement de buildings y furniture en un solo placement engine.
  5. Agregar export/import de mapas e interiores para herramientas GM.
  6. Crear backend inicial con perfiles, saves y revision number.
  7. Optimizar render cuando el mapa crezca: chunks, cache, dirty redraw.
  8. Agregar catalogo visual para colocar edificios/muebles desde UI.
